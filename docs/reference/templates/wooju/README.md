# Wooju용 OpenClaw 에이전트 템플릿

Wooju를 위한 모듈화된 에이전트 템플릿 시스템입니다. 팀의 직군, 기술 스택, 서비스 도메인에 맞게 조합하여 사용할 수 있습니다.

> **📝 이 템플릿에 대하여**
>
> 이 템플릿은 완성된 최종본이 아닌, **여러분이 팀에 맞는 에이전트를 만들기 위한 출발점**입니다.
>
> 각 팀의 상황, 컨벤션, 도메인 지식은 모두 다릅니다. 이 템플릿은 기본적인 구조와 예시를 제공하여,
> 여러분이 더 풍부하고 구체적인 내용을 추가할 수 있도록 돕기 위해 만들어졌습니다.
>
> **자유롭게 수정하고, 확장하고, 팀의 지식을 담아주세요.** 여러분의 팀이 이 템플릿을 개선하고
> 실제 업무에 맞게 발전시키는 것이 이 프로젝트의 진짜 목표입니다.

## 한 줄 요약

**보안 설정 + 당신의 직군 + 기술 스택 + 서비스를 선택하면, 그에 맞는 에이전트가 자동으로 조립됩니다.**

---

## 빠른 시작 (자동 설정)

Wooju용 에이전트는 **한 번의 명령어로 자동 설정**됩니다:

```bash
openclaw wooju
```

이 명령어는 대화형으로 다음을 수행합니다:

1. **보안 경고 확인** - API 키 사용, 샌드박스 설정, 데이터 보호 안내
2. **Docker 환경 확인** - 샌드박스 보안을 위한 Docker/Colima 설치 확인
3. **보안 프리셋 선택** - 팀 환경에 맞는 보안 수준 설정 (최고보안/중간보안/실용성)
4. **NAMC API 키 설정** - HyperCLOVA X 모델 사용을 위한 API 키 입력
5. **모델 선택** - 사용 가능한 모델 중 선택
6. **Slack 연동** - 팀 채널 연동 설정 (선택)
7. **팀 템플릿 선택** - 역할(Role) + 기술 스택(Stack) + 서비스 도메인(Service) 조합
8. **워크스페이스 자동 생성** - `~/.openclaw/wooju-workspace/` 생성 및 템플릿 파일 자동 조합
9. **샌드박스 이미지 빌드** - Docker 기반 보안 샌드박스 환경 구성
10. **게이트웨이 시작** - 에이전트 게이트웨이 실행
11. **TUI/WebUI 실행** - 터미널 UI와 웹 UI로 즉시 대화 시작

**모든 템플릿 조합은 자동으로 처리됩니다.** 아래 섹션에서 어떤 모듈이 조합되는지 확인할 수 있습니다.

## 블록 쌓기처럼 이해하기

에이전트의 성격과 능력은 **4개의 블록**을 쌓아서 만들어집니다:

```
┌───────────────────────────────────┐
│  4. 서비스 (선택)                 │  ← 지도, 쇼핑, 검색... 도메인 지식
├───────────────────────────────────┤
│  3. 기술 스택 (선택, 개발자만)    │  ← iOS, Android, 프론트엔드...
├───────────────────────────────────┤
│  2. 직군 (선택)                   │  ← 개발, 기획, 디자인, 데이터
├───────────────────────────────────┤
│  1. 공통 (필수)                   │  ← 모든 에이전트의 기본 원칙
└───────────────────────────────────┘
```

아래에서 위로 블록을 쌓으면 **나만의 에이전트**가 완성됩니다.

---

## 각 블록 설명

### 1. 공통 블록 (Core) - 모든 에이전트의 기본

모든 에이전트가 공유하는 핵심 원칙입니다. 자동으로 포함되며, 선택할 필요가 없습니다.

| 원칙 | 의미 |
|------|------|
| **Real Help** | 말만 하지 않고 직접 확인하고 결과를 가져온다 |
| **Search-First** | 되묻기 전에 스스로 찾아본다 |
| **Have an Opinion** | 이것저것 나열하지 않고 하나의 제안을 한다 |
| **Guest Mindset** | 내부 자료를 소중히 다루고, 위험한 작업 전 확인한다 |

### 2. 직군 블록 (Role) - 나의 역할

어떤 일을 하는지에 따라 에이전트의 관점이 달라집니다.

| 직군 | 에이전트가 중요시하는 것 |
|------|-------------------------|
| **개발 (Developer)** | 코드 품질, 성능, 모듈화 |
| **기획 (Planning)** | 사용자 관점, 명확한 정의, 점진적 확장 |
| **디자인 (Design)** | UX 우선, 디자인 시스템, 개발자 협업 |
| **데이터 (Data)** | 재현 가능성, 비즈니스 연결, 개인정보 보호 |

### 3. 기술 스택 블록 (Stack) - 개발자 전용

> ⚠️ **개발자 직군을 선택한 경우에만** 이 블록이 추가됩니다.

어떤 기술을 사용하는지에 따라 구체적인 기준이 달라집니다.

| 스택 | 에이전트가 알고 있는 것 |
|------|------------------------|
| **iOS** | Swift, SwiftUI, 60fps 유지, 배터리 최적화 |
| **Android** | Kotlin, Jetpack, 메모리 관리 |
| **Frontend** | TypeScript, React, 웹 성능 지표 |
| **Backend (Java)** | Spring, API 설계, 테스트 전략 |
| **DevOps** | CI/CD, 인프라, 모니터링 |

### 4. 서비스 블록 (Service) - 도메인 지식

어떤 서비스를 만드는지에 따라 도메인 전문 지식이 추가됩니다.

| 서비스 | 에이전트가 이해하는 것 |
|--------|----------------------|
| **지도 (Maps)** | 위치 서비스, 지도 렌더링, 경로 탐색 |
| **쇼핑 (Shopping)** | 상품/재고, 주문/결제, 추천 |
| **검색 (Search)** | 인덱싱, 랭킹, 자연어 처리 |

## openclaw wooju 실행 과정

터미널에서 `openclaw wooju`를 실행하면 다음 단계가 대화형으로 진행됩니다:

### 1단계: 보안 경고 확인
```
⚠️  보안 및 책임 안내
NAMC API 키 사용, 샌드박스 설정, 데이터 보호 등에 대한 안내
→ 확인 후 진행
```

### 2단계: Docker 환경 확인
```
Docker가 필요합니다 (샌드박스 보안을 위해)
→ 없으면 Colima 자동 설치 제안
→ 또는 Docker 없이 계속 (샌드박스 비활성화)
```

### 3단계: 보안 프리셋 선택
```
? 보안 수준을 선택하세요
  ○ 최고보안 (읽기전용 샌드박스, 웹 차단)
  ● 중간보안 (읽기/쓰기 샌드박스, 웹 허용) ← 기본값
  ○ 실용성 (샌드박스 없음, 웹 허용)
```

### 4단계: NAMC API 키 입력
```
? NAMC API 키를 입력하세요
→ OSS Console 또는 Commercial Console에서 발급
```

### 5단계: Slack 연동 (선택)
```
? Slack을 연동하시겠습니까?
  ● Yes
  ○ No

→ Yes 선택 시: 봇 토큰, 허용 채널 ID 입력
```

### 6단계: 템플릿 선택
```
? Select your role
  ● Developer
  ○ Planning
  ○ Design
  ○ Data

? Select your tech stack (optional)    ← 개발자만 표시됨
  ○ None
  ● iOS
  ○ Android
  ○ Frontend
  ○ Backend (Java)
  ○ DevOps

? Select your service (optional)
  ○ None
  ● Maps
  ○ Shopping
  ○ Search
```

### 7단계: 자동 구성 및 시작
```
✅ 설정 완료!
📁 설정 파일: ~/.openclaw/config.yaml
📁 환경 파일: ~/.openclaw/.env
📁 워크스페이스: ~/.openclaw/workspace/

→ 게이트웨이 자동 시작
→ TUI/WebUI 자동 실행
```

선택을 완료하면, 해당 조합에 맞는 에이전트 설정 파일들이 자동으로 생성되고 즉시 사용할 수 있습니다.

## 4-Layer 모듈 시스템

```
wooju/
├── base/                    # Layer 0: 모든 팀 공통 (필수)
│   ├── SOUL.base.md         # 핵심 원칙
│   ├── AGENTS.base.md       # 워크스페이스 운영
│   └── SECURITY.base.md     # 보안 정책
│
├── roles/                   # Layer 1: 직군별
│   ├── developer/           # 개발자
│   ├── planning/            # 기획/PM
│   ├── design/              # 디자인
│   └── data/                # 데이터 분석
│
├── stacks/                  # Layer 2: 기술 스택별
│   ├── ios/                 # iOS 네이티브
│   ├── android/             # Android 네이티브
│   ├── frontend/            # 웹 프론트엔드
│   ├── backend-java/        # Java/Kotlin 백엔드
│   └── devops/              # DevOps/SRE
│
├── services/                # Layer 3: 서비스 도메인별
│   ├── maps/                # 지도
│   ├── shopping/            # 쇼핑
│   └── search/              # 검색
│
└── examples/                # 조합 예시 (참고용)
    ├── maps-ios/            # 지도 iOS 팀
    ├── shopping-fe/         # 쇼핑 FE 팀
    └── search-be/           # 검색 BE 팀
```

## 생성되는 파일들

onboard가 완료되면 워크스페이스에 다음 파일들이 만들어집니다:

| 파일 | 역할 |
|------|------|
| `SOUL.md` | 에이전트의 핵심 원칙과 가치 (위 블록들이 합쳐진 결과) |
| `IDENTITY.md` | 에이전트의 역할, 권한, 전문 분야 |
| `USER.md` | 사용자(팀)의 선호와 컨벤션 |
| `AGENTS.md` | 세션 운영 방식, 메모리 관리 |

이 파일들은 에이전트가 매 대화 시작 시 읽어서 "나는 누구이고, 어떻게 행동해야 하는지"를 파악합니다.

---

## 커스터마이징

생성된 파일들은 시작점입니다. 팀의 상황에 맞게 자유롭게 수정하세요:

- **팀 컨벤션 추가:** 코드 리뷰 규칙, 커밋 메시지 형식 등
- **용어 사전 링크:** 팀 위키, Confluence 문서
- **추가 도구:** 팀에서 사용하는 특수 도구나 스크립트

에이전트는 이 파일들을 읽고 학습하므로, 파일을 업데이트하면 에이전트의 행동도 변합니다.

## 고급: 수동 커스터마이징

> **💡 참고:** 대부분의 경우 `openclaw wooju` 명령어로 자동 설정하는 것을 권장합니다.
> 이 섹션은 템플릿을 직접 조합하거나 세밀한 커스터마이징이 필요한 경우에만 참고하세요.

### 1단계: 워크스페이스 생성

```bash
mkdir -p ~/my-agent-workspace
cd ~/my-agent-workspace
```

### 2단계: 모듈 선택

팀에 맞는 조합을 선택합니다:

| 팀 | 조합 |
|----|------|
| 지도 iOS | base + developer + ios + maps |
| 쇼핑 FE | base + developer + frontend + shopping |
| 검색 BE | base + developer + backend-java + search |
| 기획팀 | base + planning + (서비스) |

### 3단계: 파일 조합

#### AGENTS.md (필수)

Base의 AGENTS.base.md를 복사합니다:

```bash
cp docs/reference/templates/wooju/base/AGENTS.base.md ~/my-agent-workspace/AGENTS.md
```

#### SOUL.md 조합

아래 순서대로 섹션을 조합합니다:

```markdown
# SOUL.md

<!-- 1. Base: 핵심 원칙 (필수) -->
<!-- SOUL.base.md 내용 -->

<!-- 2. Role: 직군별 가치 -->
<!-- roles/developer/SOUL.developer.md 또는 다른 role -->

<!-- 3. Stack: 기술 스택 기준 -->
<!-- stacks/ios/SOUL.ios.md 또는 다른 stack -->

<!-- 4. Service: 도메인 지식 -->
<!-- services/maps/DOMAIN.maps.md 또는 다른 service -->
```

#### IDENTITY.md, USER.md

팀에 맞게 작성합니다. `roles/` 아래의 템플릿을 참고하세요.

#### SOUL.md 섹션 구조

최종 SOUL.md는 다음 구조를 권장합니다:

```markdown
# SOUL.md: [팀명] [역할]

## 1. 정체성 (Identity)
<!-- Core + Role에서 -->

## 2. 핵심 원칙 (Core Principles)
<!-- SOUL.base.md에서 -->

## 3. [직군] 가치 (Role Values)
<!-- SOUL.developer.md 등에서 -->

## 4. 기술 기준 (Technical Standards)
<!-- SOUL.ios.md 등에서 -->

## 5. 도메인 지식 (Domain Knowledge)
<!-- DOMAIN.maps.md 등에서 -->

## 6. 커뮤니케이션 (Communication)
<!-- Core + Role에서 -->

## 7. 자아 업데이트 (Self-Evolution)
<!-- Core에서 -->
```

### 4단계: 팀별 추가 규칙

각 레이어의 파일 하단에 팀별 규칙을 추가할 수 있습니다:

```markdown
## Team-Specific Rules

- 우리 팀은 PR 리뷰 시 최소 2명의 승인을 요구함
- 모든 커밋은 Jira 이슈 번호를 포함해야 함
- ...
```

### 5단계: 새로운 서비스/스택 추가

1. `services/` 또는 `stacks/` 아래에 새 디렉토리 생성
2. 기존 템플릿을 참고하여 `DOMAIN.*.md` 또는 `SOUL.*.md` 작성
3. `examples/`에 조합 예시 추가 (선택)

## FAQ

**Q: 직군만 선택하고 서비스는 선택 안 해도 되나요?**

A: 네, 서비스와 스택은 선택사항입니다. "개발자"만 선택해도 기본적인 개발 원칙을 갖춘 에이전트가 생성됩니다.

**Q: 나중에 서비스를 추가하고 싶으면?**

A: `openclaw wooju`를 다시 실행하거나, `docs/reference/templates/wooju/services/`에서 해당 서비스 파일을 복사해 `~/.openclaw/workspace/SOUL.md`에 추가하면 됩니다.

**Q: 우리 팀 서비스가 목록에 없어요.**

A: `services/` 폴더에 새 서비스를 추가할 수 있습니다. 기존 서비스 파일(`DOMAIN.maps.md` 등)을 참고해서 `DOMAIN.myservice.md`를 만들어보세요.

**Q: 개발자인데 여러 스택을 다루면요?**

A: 가장 주력으로 사용하는 스택 하나를 선택하세요. 이후 `SOUL.md`에 다른 스택 관련 내용을 직접 추가할 수 있습니다.

**Q: 모든 레이어를 다 사용해야 하나요?**

A: Base(Layer 0)만 필수입니다. 나머지는 팀 상황에 맞게 선택하세요.

**Q: 기존에 만든 템플릿이 있는데요?**

A: 기존 템플릿의 내용을 이 구조로 재배치하면 됩니다. 특히 도메인 지식은 `services/`에 정리하면 다른 팀에서도 재사용할 수 있습니다.

**Q: 비개발 직군도 사용할 수 있나요?**

A: 네. `roles/planning`, `roles/design`, `roles/data` 등 비개발 직군용 템플릿도 준비되어 있습니다. (일부는 준비 중)
